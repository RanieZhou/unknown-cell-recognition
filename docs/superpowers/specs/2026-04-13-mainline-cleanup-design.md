# scGRN 主线收口设计

## 背景

当前主线已经完成第一批接口与契约修复，但仍存在四类会持续制造混淆的问题：

1. `src/scgrn` 内仍残留 `E00X` 这类历史实验编号语义。
2. 主线仍保留 `scgpt` 预设代码，但当前并无该需求。
3. 若干 fallback 会在不改变报错状态的前提下改变结果语义，削弱可复现性。
4. 方法层仍同时保留固定权重、双阈值策略等多条口径，主线不够唯一。

本次设计的目标不是增加新功能，而是把当前代码库收口成一条唯一、清晰、可复现的主线。

## 目标

### 1. 主线语义唯一

- `src/scgrn` 不再出现 `E00X` 等历史实验编号。
- 主线不再保留没有现实需求的伪扩展点，例如 `scgpt`。
- 对外字段名、报告名、日志文案直接使用当前主线语义。

### 2. 运行行为可解释

- 根目录脚本继续作为 thin CLI entrypoints。
- 真正实现继续集中在 `src/scgrn`。
- 主线默认不再 silent fallback，也不再 implicit recompute。

### 3. 最终结果口径唯一

- 只保留一个最终阈值策略。
- 融合权重不再依赖手工固定值。
- backbone 间允许内部实现不同，但下游契约保持统一。

## 非目标

- 不在这次收口中引入新的 backbone。
- 不把主线改造成新的黑盒二级模型。
- 不为历史实验路径继续扩展兼容分支。
- 不在本轮同时重构整个 rescue 方法族，只收口当前主线使用的那一条。

## 范围

本次设计覆盖以下四个模块。

### 模块 A：命名与主线清理

#### 目标

清除 `src/scgrn` 中所有 `E00X` 和“旧实验步骤名”残留，把主线公共语义层改成当前方法语义。

#### 清理范围

- 代码中的列名、字段名、常量名。
- 注释、报告标题、日志文案。
- 中间产物 schema。
- rescue、threshold、evaluation 中仍带实验阶段编号的命名。

#### 设计要求

- 主线内部代码和新产物不再出现 `E00X`。
- 若逻辑已经进入主线，就直接以主线名义存在，不保留“从旧实验搬来”的痕迹式命名。
- 改名必须按 schema 级变更处理，而不是仅做字符串替换。

#### 兼容策略

- 新生成的主线产物全部使用新字段名。
- 若需要读取旧结果，只允许在边缘读取层做一次性映射。
- 映射后立即转成新字段，不允许旧字段继续流入主线核心逻辑。

#### 验收标准

- `src/scgrn` 中不再出现 `E00X`。
- 主线核心产物不再带实验编号字段名。
- 旧字段只出现在兼容读取边缘层。

### 模块 B：支持面收缩

#### 目标

物理删除 `scgpt` 相关预设代码，让主线支持集合只保留当前真实维护的 backbone。

#### 设计要求

- 删除常量中的 `scgpt` 支持声明。
- 删除 config 校验中的相关分支和占位逻辑。
- 删除 registry 中的 `scgpt` 分发路径。
- 删除文档、注释、报错文案中“未来可能支持 `scgpt`”的主线表述。

#### 不保留的内容

- `UNIMPLEMENTED_BACKBONES`
- `NotImplementedError("scgpt ...")`
- “reserved for future extension” 这类主线噪声

#### 验收标准

- 主线支持集合与代码、文档、CLI 说明完全一致。
- 用户不再看到“看起来支持但实际不能用”的 backbone 入口。

### 模块 C：可复现性优先的 fail-fast 收紧

#### 目标

把主线默认行为从“尽量继续跑”改成“优先保持结果定义稳定”。

#### fallback 分类与处理

##### C1. 工件契约缺失

例如训练后应复用的 split 文件缺失，或某阶段依赖的中间产物缺失。

处理原则：直接报错，不自动补算。

原因：这类问题属于 pipeline contract 破坏。自动补算会改变实验状态来源，削弱结果可复现性。

##### C2. 数值策略降级

例如 Mahalanobis 失败后自动回退 Euclidean。

处理原则：主线默认报错，不自动降级。

原因：距离度量一旦变化，方法定义已经变化。这不属于普通工程兜底。

若后续确有研究需要，可单独设计显式配置开关，但不属于整理后的默认主线。

##### C3. 历史兼容读取

例如旧结果文件字段名不同，或旧版本输出结构略有差异。

处理原则：仅允许保留在读取旧结果的边缘层，并显式 warning。

原因：这是兼容旧结果，不是主线训练、推理、救援方法的一部分。

#### 默认主线规则

- 缺失依赖工件：报错。
- 数值方法失败：报错。
- 旧结果兼容读取：warning + 边缘映射。
- 不再 silent fallback。
- 不再 implicit recompute。

#### 验收标准

- 主线训练、推理、救援链路中不再存在 silent fallback。
- 同一 run 不会因为缺工件而悄悄走另一条路径。
- 报错能直接暴露真实失败点。

### 模块 D：方法主线单一化

#### D1. `scnym` adapter 结构审查

##### 目标

先判断 `scnym` 是否只是复用了统一接口，还是被 `scanvi` 的内部假设错误绑住。

##### 两层审查

第一层是接口层复用是否合理：

- 统一 adapter 接口。
- 统一接入 `train / predict / build_expression_artifacts`。
- 统一输出下游所需 schema。

这部分通常合理，应保留。

第二层是内部假设是否被错误共享：

- `scnym` 的概率输出是否被当成与 `scanvi` 同性质处理。
- calibration、confidence、distance 的构造是否默认依赖 `scanvi` 的分布特性。
- expression artifact 的若干字段是否对 `scanvi` 自然、对 `scnym` 只是为了兼容接口硬凑。

##### 可能结论

- 仅接口统一，无内部误绑：保留现状，只做小修。
- artifact 构造层轻度误绑：允许 `scnym` 与 `scanvi` 在构造层分叉，但保持统一 schema。
- confidence / distance 假设整体不合适：把 `scnym` 当成独立 backbone 路径设计 artifact，再映射到统一下游接口。

##### 设计结论

这次收口先做结构审查，不承诺立刻重写 `scnym`。统一的是下游 schema，不是上游实现细节。

#### D2. expression 侧权重从手工固定改为学习式组合

##### 当前问题

固定 `0.5/0.5` 的问题不只是可能不最优，更在于它缺少数据支持，并把一种方法选择伪装成默认事实。

##### 候选方案

1. 继续手动固定权重。
2. 只在 expression 内部学习组合权重。
3. 直接把 expression 与 GRN 各分数一起放进逻辑回归，学习最终 unknown score。

##### 推荐方案

默认主线采用方案 2：只在 expression 内部学习组合权重。

原因：

- 改动聚焦。
- 解释链清楚。
- 保留 expression 与 GRN 的分层含义。
- 避免把当前主线直接改成新的黑盒二级模型。

##### 不作为默认主线的方案

“把 expression + GRN 全部分数直接送入统一逻辑回归”在实验上可能可行，但不适合作为这次整理后的默认主线。若后续证明确有显著收益，应作为单独方法分支评估，而不是直接并入当前唯一主线。

#### D3. 最终阈值策略只保留 `globalT`

##### 结论

整理后的默认主线只保留 `globalT`，删除 `bucketT`。

##### 原因

`bucketT` 会显著提高主线复杂度。它依赖：

- bucket 划分稳定。
- 各 bucket 样本量充分。
- 各 bucket 校准可靠。

这会让最终结果解释变成“先分桶，再按桶阈值判定”。对当前“唯一主线、可解释、可维护”的目标不利。

`globalT` 更适合作为整理后的唯一主结果，因为它：

- 结果定义更单一。
- 校准流程更直接。
- 更易文档化。
- 更易跨数据集维护。

##### 删除要求

不是只隐藏 `bucketT` 输出，而是彻底删除：

- `bucketT` 预测路径。
- 相关阈值拟合辅助函数。
- 相关报告与输出字段。
- README 和文档中的双策略说明。
- 相关测试与命名。

## 实施顺序

### 第 1 批：主线纯化

- 清除 `E00X` 与历史实验语义。
- 删除 `scgpt`。
- 明确入口层只是 thin wrapper。
- 更新 README、`CLAUDE.md` 与对外 schema 命名。

### 第 2 批：可复现性收紧

- 缺工件直接报错。
- 数值策略失败默认报错。
- 移除主线 silent fallback。
- 把历史兼容逻辑收缩到边缘读取层。

### 第 3 批：方法主线单一化

- 审查 `scnym` 结构适配是否失真。
- 把 expression 内部固定权重改成 calibration-set 学习式组合。
- 只保留 `globalT`。
- 删除 `bucketT`。

## 验证要求

### 命名与支持面

- 搜索 `src/scgrn`，确认无 `E00X`。
- 搜索主线代码和文档，确认无 `scgpt`。
- 检查根目录入口、README、`CLAUDE.md` 与实际支持集合一致。

### 可复现性

- 缺失 split 或中间工件时应直接失败。
- 数值方法失败时应直接失败，不再悄悄改用其他度量。
- 历史兼容读取时应能看到显式 warning。

### 方法主线

- `scnym` 的 artifact 构造是否仍被 `scanvi` 的内部假设绑住，应有明确结论。
- expression 权重不再是硬编码固定值。
- 最终结果与报告中只保留 `globalT`，不再出现 `bucketT`。

## 风险与取舍

### 1. 报错会变多

这是预期变化。此次收口优先保证结果定义稳定，而不是继续容忍“能跑但路径不确定”的行为。

### 2. 中间产物 schema 会变更

这也是预期变化。字段名从历史实验编号改成主线语义后，测试、文档、兼容读取层需要同步调整。

### 3. `scnym` 不一定立即重写

本设计先要求结构审查。只有证据表明它被 `scanvi` 的内部假设错误绑住，才进入局部分叉或重构。

### 4. 动态权重不追求一步到位

这次只把默认主线推进到“expression 内部学习式组合”。不把整个主线直接改造成统一逻辑回归终判器。

## 最终状态

完成后，整理后的主线应满足以下条件：

- 主线代码不再带历史实验编号语义。
- 主线不再保留无需求的 backbone 占位。
- 默认运行行为可复现、可解释、可报错。
- `scnym` 是否适配失真有清晰结论。
- expression 权重不再靠手工固定值。
- 最终结果只保留一个阈值口径，即 `globalT`。
- README、`CLAUDE.md`、测试、输出 schema 与主线实现一致。
